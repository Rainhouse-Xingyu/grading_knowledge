# 大连东软信息学院项目双轨制AI评分系统——技术设计文档 (V3-教师管控版)

## 1. 系统架构与技术栈选型

本系统采用**前后端分离**及**双轨制服务**架构。为了完美契合“学生只管提交、教师统一管控大模型算力”的教学实际需求，系统采用 **Java Spring Boot（主业务） + Redis（状态流转与并发锁） + Python FastAPI（AI/OCR算法服务）** 的核心架构。

| 层面 | 选型技术/工具 | 说明 |
| :--- | :--- | :--- |
| **前端门户** | Vue 3 + Vite + Element Plus + Pinia | 选用东软主流技术栈，模块化组件开发，构建教师批阅工作台与学生提交面板。 |
| **富文本/预览**| `v-md-editor` + `pdfjs-dist` | 用于教师端在线无缝调整Markdown评语，并支持向学生端输出并渲染PDF报告。 |
| **后端主服务** | Java Spring Boot 3.x + Spring Security | 负责核心业务逻辑、学校CAS单点登录对接、文件安全路由及物理地址映射。 |
| **分布式缓存/控制**| **Redis 7.x + Redisson** | **核心变更：管理学生无感知上传幂等、锁定教师触发的批量AI任务、追踪耗时异步AI状态。** |
| **AI/算法服务** | Python FastAPI + LangChain / LlamaIndex | 专门对接云端DeepSeek API、本地Ollama以及OCR模型，Python生态提供更高的AI适配度。 |
| **关系型数据库**| MySQL 8.0 | 存储持久化数据：用户基础信息、文件物理路径映射关系、三阶段分数、教师修改评语。 |
| **向量数据库** | **Milvus** (或 Qdrant) | 存储教师上传的各课程评分标准文档切片及向量数据，用于AI评阅时的RAG知识检索。 |
| **对象存储** | MinIO (私有化本地部署) | 本地化安全存储：学生提交的源码压缩包、图文报告原件、AI及教师最终生成的报告。 |

## 2. Redis 核心应用场景与详细设计（教师触发架构微调）

将大模型的触发权收回到教师端后，Redis 的核心职责从“防范学生刷量”转变为“保障教师端大并发批阅稳定、控制学校算力配额”。

### 2.1 学生端：轻量级“文件上传幂等锁”
* **业务场景**：学生连续猛击“提交作业”或因网络卡顿产生多条多线程网络请求。
* **Redis 设计**：纯业务层面的轻量级分布式锁，仅用于防范 MinIO 产生垃圾重复文件和 MySQL 插入两条相同阶段的记录。
* **Key 设计**：`neusoft:lock:student_upload:{student_no}:{course_id}:{stage_num}`
* **执行策略**：`tryLock(wait_time=3s, lease_time=10s)`。一旦学生的文件成功写入 MinIO 且 MySQL 状态更新为 `0（已提交待评测）`，锁立刻释放。学生此时完成提交，不触发任何 AI 操作。

### 2.2 教师端：重度耗时“AI 评审触发锁”（核心变更）
* **业务场景**：教师在后台点击“批量评审全班”或对某个特定学生点击“触发AI评测”时，网络卡顿可能导致教师重复点击，或者同组的协同教师重复触发同一个学生的同一阶段作业。
* **Redis 设计**：基于 **Redisson** 的长生存期强互斥锁。如果 AI 正在跑某个学生的作业，任何人都无法再次对该学生发起评审。
* **Key 设计**：`neusoft:lock:teacher_eval:{student_no}:{course_id}:{stage_num}`
* **执行策略**：`tryLock(wait_time=5s, lease_time=180s)`（考虑 AI+OCR 的链路耗时，给一个较长的自动过期时间防止系统死锁）。重复请求直接拒绝，提示：“该学生的作业正在进行 AI 评测中，请勿重复触发”。

### 2.3 大模型算力配额控制与自适应降级切换
* **配额计数器**：在 Redis 中维护一个每日配额计数器 `neusoft:quota:deepseek:date:{yyyyMMdd}`。教师触发批量评审时，系统会优先使用该计数器，避免全班作业瞬间压垮学校购入的云端 DeepSeek 企业配额。
* **自适应降级**：当触发云端限流或配额耗尽时，后端捕获异常并在 Redis 中写入降级标识 `neusoft:config:model_fallback = "local"`。FastAPI 算法服务识别到该状态后，自动路由至本地 **Ollama** 驱动的本地私有化模型（如 `DeepSeek-R1-Distill-Qwen-14B`），确保批阅任务不间断。

### 2.4 异步 AI 评测任务状态流转缓存（面向教师端）
* **业务场景**：教师启动评测后，页面需要实时感知全班学生的批改进度。
* **Redis 设计**：采用异步任务队列模式，将任务状态维护在 Redis 中，教师端前端通过 `task_id` 实施轮询。
* **Key 设计**：`neusoft:task:status:{task_id}`
* **状态机设计**：
  * `10-等待队列中` -> `20-PaddleOCR图文解析中` -> `30-Milvus评分标准检索中` -> `40-DeepSeek深度分析中` -> `50-完成`。教师工作台可据此渲染出清晰的“多作业排队进度条”。

## 3. 核心业务功能模块细化设计

### 模块一：统一身份认证（CAS登录）
* 用户通过学校 CAS 换取 Ticket 成功后，后端生成自定义 `Access-Token` 并写入 Redis 保持会话。同时将 CAS 返回的学生9项字段或教师6项字段持久化写入 MySQL 对应表中。

### 模块二：静态文件存储与完全物理隔离
* **学生只负责文件上传**。前端将 `.zip` 压缩包和报告上传至后端后，后端通过 UUID 混淆物理路径并存入 MySQL 映射表。向前端隐藏一切真实文件目录。只有当教师端或学生端（解锁后）发起下载请求时，后端校验 Redis 中的 Token 权限后，再从 MinIO 读取文件以流的形式写回。

### 模块三：图文报告解析与 OCR 模块
* **触发时机移至教师端**。当教师点击评审后，FastAPI 算法服务才开始解压文件。利用 `PaddleOCR` 识别学生报告中的图表、实验控制台截图中的文字代码，将其转化为结构化文本，按段落原位拼接到主报告文本中。

### 模块四：三阶段分段式检测与联合评审
* **联合评审逻辑**：若教师在触发评审时勾选了“联合评审”，FastAPI 服务不仅会从 Milvus 检索当前的评分标准文档，还会去 MySQL 读取该学生**上一个阶段被教师最终审核锁定的评语、修改记录和分数**作为上下文。通过专门设计的 System Prompt，要求 DeepSeek 严格审查本次提交相比上次是否有“实质性的工作增量”，防止学生提交成品或不做作业。

### 模块五：教师动态调分、在线预览与一键下发（核心控制面）
* **教师工作台**：AI 跑完后，数据持久化到 MySQL，状态变为 `2-AI评测完成(待发布)`。此时学生端处于完全屏蔽状态。
* **在线微调**：教师在线打开 `v-md-editor`，直接预览 AI 生成的原始 Markdown 报告。教师可以对 AI 的幻觉、误判部分进行在线文本修改，并在框中输入最终评定的分数（覆盖 AI 的严格分）。
* **一键发布**：教师点击“确认下发”后，状态更改为 `3-已下发`。此时学生端页面解锁，学生才能看到最终成绩，在线预览并一键导出为 PDF 评审报告。

## 4. 向量数据库设计（Milvus）

* **向量数据库名称**：`neusoft_grading_knowledge`
* **文本向量化模型（Embedding）**：`bge-large-zh-v1.5`（1024维）。

### 集合（Collection）结构设计：`grading_standards`

| 字段名 (Field) | 数据类型 (Type) | 是否主键 | 约束/索引 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `chunk_id` | INT64 | 是 | 自动递增 | 唯一切片ID |
| `vector` | FLOAT_VECTOR | 否 | **1024 维** | 评分标准文本切片转换后的密集向量 |
| `course_id` | VARCHAR | 否 | 长度 32 (标量索引) | 课程项目唯一ID，用于在多课并发时做物理多租户数据隔离 |
| `teacher_id`| VARCHAR | 否 | 长度 32 | 上传该评分标准的教师工号 |
| `stage` | INT32 | 否 | 标量索引 | 适用阶段（0:通用标准; 1:阶段一; 2:阶段二; 3:阶段三） |
| `content` | VARCHAR | 否 | 长度 4000 | 原始评分规则、扣分要点文本明文 |
